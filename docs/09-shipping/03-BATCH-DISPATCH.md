# Shipping — Batch Dispatch Flow

A dispatch batch allows multiple shipping marks to be prepared offline and then submitted to
the carrier in a single coordinated operation. This is the primary workflow for bulk
fulfilment runs.

---

## Two-Phase Overview

```
Phase 1 — Draft (accumulate)         Phase 2 — Close (submit)
─────────────────────────────────    ──────────────────────────────────────────
 POST /shipping/batches               PATCH /shipping/batches/{id}/close
   → OPEN batch                         for each QUOTED mark:
                                           materializer.Materialize(&mark)
 POST /batches/{id}/marks                    ├── carrier API call
   → mark = QUOTED, linked                   ├── success → CREATED
                                             └── failure → FAILED
 ... add more marks ...                  batch → CLOSED
                                         manifest cache invalidated
 DELETE /batches/{id}/marks/{m}          event: BatchClosed
   → mark permanently deleted
```

---

## Phase 1 — Drafting Marks

### Create a batch

```
POST /shipping/batches
{ "carrierId": "tcc", "createdBy": "ops-maria" }
```

A new `DispatchBatch` is created with `status = OPEN`. All marks added to this batch must
belong to the same carrier.

### Add (draft) a mark

```
POST /shipping/batches/{id}/marks
{
  "orderId": "order-uuid",
  "quotationId": "q-uuid",        ← optional; captures QuotedFreightCost snapshot
  "shipmentMode": "parcel",
  "sender": { ... },
  "recipient": { ... },
  "units": [ ... ],
  "declaredValue": 150000,
  "collectOnDeliveryAmount": 0
}
```

The mark is created with `status = QUOTED` and assigned `DispatchBatchID`. A `DraftSnapshot`
(JSON of all mark fields) is captured at this point. Duplicate drafts for the same order
within the same batch are allowed — operators must manage duplicates manually.

**Validation enforced:**
- Batch must be `OPEN`.
- Mark's `CarrierID` must equal the batch's `CarrierID`.

### Remove a drafted mark

```
DELETE /shipping/batches/{id}/marks/{markId}
```

Permanently deletes the mark (hard delete, not soft delete). Only `QUOTED` marks may be
removed. Closed batches do not accept removals.

---

## Phase 2 — Closing the Batch

```
PATCH /shipping/batches/{id}/close
```

### What happens during close

1. All `QUOTED` marks belonging to the batch are fetched.
2. For **each mark** (in sequence):
   a. `materializer.Materialize(&mark)` is called.
   b. Internally calls `provider.GenerateMark(ctx, &mark)` on the TCC REST API.
   c. **Success path:**
      - `mark.TrackingNumber` = carrier tracking number
      - `mark.DocumentRef` = label URL
      - `mark.Status` = `CREATED`
      - Repository updated.
      - `shipping.v1.mark.generated` event (or `mark.created`) published.
   d. **Failure path:**
      - `mark.FailureReason` = carrier error message
      - `mark.Status` = `FAILED`
      - Repository updated.
      - `shipping.v1.mark.failed` event published.
3. Individual failures **do not abort the close** — remaining marks continue to be
   processed. Partial batches (some CREATED, some FAILED) are normal and expected when
   carrier validation rejects individual shipments.
4. Batch `status` = `CLOSED`, `ClosedAt` = now.
5. Manifest cache invalidated (Redis).
6. `shipping.v1.batch.closed` event published with `{marksCreated, marksFailed}` counters.

### No automatic rollback

There is **no rollback mechanism** for a closed batch. FAILED marks remain associated with
the batch and can be inspected via `GET /shipping/batches/{id}`. Voiding (`PATCH
/shipping/marks/{id}/void`) changes the local status only — no carrier cancellation API is
called. Out-of-band carrier cancellation is the operator's responsibility.

## Best Active Mark for an Order

When multiple marks exist for the same order (e.g., a reattempt after a failure), the system
resolves the "active" mark using a priority scheme:

```
GET /shipping/orders/{orderID}/dispatch
```

| Priority | Status |
|---------|--------|
| 3 (highest) | `QUOTED` |
| 2 | `CREATED` |
| 1 | `GENERATED` |
| — (excluded) | `FAILED`, `VOIDED`, `REMOVED` |

The mark with the highest priority status is returned. If multiple marks share the same
priority, the most recently created one wins.

---

## Manifest Document

After a batch is closed, a merged PDF manifest can be retrieved:

```
GET /shipping/batches/{id}/manifest-document
```

The manifest PDF is generated by:
1. Downloading each mark's label PDF from `DocumentRef`.
2. Merging all label pages into a single PDF using `pdfcpu`.
3. Appending a summary page (batch ID, carrier, close date, mark count) generated with
   `gofpdf`.

The result is cached in Redis with configurable TTL. Subsequent requests within the TTL
window return the cached PDF without re-generating.

## Full Example: Bulk Dispatch Run

```
ops-maria
  │
  ├─ POST /shipping/batches
  │   { carrierId: "tcc", createdBy: "ops-maria" }
  │   ← batch: { id: "b-001", status: "OPEN", markIDs: [] }
  │
  ├─ POST /shipping/batches/b-001/marks   (order-A)
  │   ← mark: { id: "m-001", status: "QUOTED", orderId: "order-A" }
  │
  ├─ POST /shipping/batches/b-001/marks   (order-B)
  │   ← mark: { id: "m-002", status: "QUOTED", orderId: "order-B" }
  │
  ├─ POST /shipping/batches/b-001/marks   (order-C — wrong address)
  │   ← mark: { id: "m-003", status: "QUOTED", orderId: "order-C" }
  │
  │   [ops-maria reviews and removes the bad mark]
  │
  ├─ DELETE /shipping/batches/b-001/marks/m-003
  │   ← mark m-003 permanently deleted
  │
  ├─ PATCH /shipping/batches/b-001/close
  │   Processing:
  │     m-001 → TCC API → { numeroremesa: "TCC-111", urlguia: "https://..." }
  │                  → status = CREATED ✅
  │     m-002 → TCC API → { error: "invalid destination" }
  │                  → status = FAILED, failureReason = "invalid destination" ❌
  │
  │   Batch: { status: "CLOSED", closedAt: "2026-03-28T15:00:00Z" }
  │   Event: shipping.v1.batch.closed { marksCreated: 1, marksFailed: 1 }
  │
  └─ GET /shipping/batches/b-001/manifest-document
      ← merged PDF for successful marks
```
